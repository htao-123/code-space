```md
【角色结论】
- 用一句话总结根因层级和最小修复方向。

【已核实输入】
- 当前角色：方案设计师
- 当前角色标识：solution-designer
- 当前交接标识：
- 需求标识：
- 项目落点：
- 下一角色标识：implementer
- 问题类型：
- 复现步骤：
- 预期结果：
- 实际结果：
- 根因摘要：
- 原始需求：
- architect handoff：
- code-investigator handoff：
- 当前任务文档：

【调研发现】
- 已验证的内部事实：
- 是否需要外部调研：
- 外部调研来源：
- 外部调研结论：
- 内部证据清单：例如 `EVID-IN-001 -> path/to/file`
- 外部证据清单：例如 `EVID-EX-001 -> references/external/doc-snapshot.md | https://example.com/doc`
- 事实清单：使用 `FACT-001 -> 证据摘录：摘录内容` 格式，不夹带额外结论
- 证据映射：例如 `FACT-001 -> EVID-IN-001::keyword::摘录`
- 推断说明：
- 未验证项：

【交付物】
- 【根因分析】
  - 问题发生层:
  - 为什么在这里:
- 【解决方案】
  - 修改点 1:
  - 修改方式:
  - 修改点 2:
  - 修改方式:
- 【影响范围】
  - 受影响模块/流程:
- 【兜底策略】
  - 异常处理:
  - 回退策略:

【约束】
- 只允许在批准的文件范围内实现
- 不允许新增未定义接口
- 不允许顺手重构

【校验标准】
- 说明了为什么这么改
- 覆盖异常或失败情况
- 影响范围明确列出

【禁止事项】
- 不写代码
- 不改变前序角色范围
- 不增加未请求功能

【交接给下一个角色】
- 下一角色：开发者【负责严格按方案实现，不扩范围不重设计】
- 可用输入：原始需求 + 本 handoff + 批准的文件范围
- 非目标：重设计、范围扩张
- 完成条件：下一角色已拿到明确修改点、影响范围与兜底策略
```
