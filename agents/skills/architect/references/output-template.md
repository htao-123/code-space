# Architect Output Template

```md
【角色结论】
- 用一句话总结改动应该落在哪些模块和层级。

【已核实输入】
- 当前角色：建筑师
- 当前角色标识：architect
- 当前交接标识：
- 需求标识：
- 项目落点：
- 下一角色标识：code-investigator
- 问题类型：
- 复现步骤：
- 预期结果：
- 实际结果：
- 原始需求：
- requirement handoff：
- 当前任务文档：
- 若为已有项目且文档缺失，补录文档：

【调研发现】
- 当前项目结构事实：
- 已确认的目录落点：
- 与本轮改动相关的模块边界：
- 历史数据结构状态：
- 迁移脚本要求：
- 是否存在历史数据与新版本数据结构不一致：
- 若存在，迁移需求判断：
- 若触发文档补录，补录后的已知/未知边界：
- 是否需要外部调研：
- 外部调研来源：
- 外部调研结论：
- 内部证据清单：例如 `EVID-IN-001 -> path/to/file`
- 外部证据清单：例如 `EVID-EX-001 -> references/external/doc-snapshot.md | https://example.com/doc`
- 事实清单：使用 `FACT-001 -> 证据摘录：摘录内容` 格式，不夹带额外结论
- 证据映射：例如 `FACT-001 -> EVID-IN-001::keyword::摘录`
- 推断说明：
- 未验证项：

【交付物】
- 【涉及模块】
  - 文件/模块 1:
  - 职责:
  - 文件/模块 2:
  - 职责:
- 【数据流】
  - 输入 -> 处理 -> 输出
- 【改动策略】
  - 新增 / 修改 / 复用:
  - 数据迁移 / 兼容处理取舍:
- 【不变量】
  - 不能破坏的结构 1:
  - 不能破坏的结构 2:

【约束】
- 不允许修改无关模块
- 不允许新增架构层
- 历史数据与新版本结构不一致时，默认走迁移脚本，不默认增加长期兼容分支

【校验标准】
- 改动范围最小
- 方案符合现有架构
- 每个涉及模块都有明确理由

【禁止事项】
- 不给代码 diff
- 不替代侦查员做事实推断
- 不超出需求范围变更行为

【交接给下一个角色】
- 下一角色：侦查员【负责收集代码事实、调用链与现有实现证据】
- 可用输入：原始需求 + requirement handoff + 本 handoff
- 非目标：方案设计、编码实现
- 完成条件：下一角色已拿到明确的模块范围和结构约束
```
