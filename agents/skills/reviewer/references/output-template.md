# Reviewer Output Template

```md
【角色结论】
- 用一句话总结实现是否通过审查，以及最大风险是什么。

【已核实输入】
- 当前角色：审核员
- 当前角色标识：reviewer
- 当前交接标识：
- 需求标识：
- 下一角色标识：tester
- 问题类型：
- 复现步骤：
- 预期结果：
- 实际结果：
- 根因摘要：
- 回归检查范围：
- 原始需求：
- solution handoff：
- implementer handoff：
- 当前任务文档：

【调研发现】
- 内部调研范围：
- 内部调研结论：
- 外部调研范围：
- 已检查改动范围：
- 已核实风险上下文：
- 外部调研来源：
- 官方事实结论：
- 主流方案结论：
- 最佳实现参考结论：
- 是否发现新增外部差异：
- 推断说明：
- 未验证项：
- 需要用户确认：
- 推荐方案：
- 推荐原因：
- 主要权衡：

【交付物】
- 【业务正确性检查】
  - 是否满足需求:
  - 是否存在漏场景:
  - 是否存在需求理解偏差:
- 【范围与越界检查】
  - 是否为最小必要改动:
  - 是否存在无关修改:
  - 是否存在顺手重构/额外抽象:
- 【逻辑与可读性检查】
  - 命名是否准确:
  - 分支/状态流是否清晰:
  - 是否存在过长方法/魔法值/深层嵌套:
- 【架构边界检查】
  - 是否破坏分层:
  - 是否存在越层逻辑:
  - 是否破坏模块职责:
- 【状态与数据流检查】
  - 数据来源/去向是否清晰:
  - 是否存在状态不同步风险:
  - 是否存在异步覆盖/销毁后更新风险:
- 【异常与兜底检查】
  - 失败路径是否覆盖:
  - 错误提示是否合理:
  - 是否存在吞异常/状态不一致风险:
- 【性能与资源检查】
  - 是否存在重复请求/重计算:
  - 是否存在监听器/订阅/定时器未释放:
  - 是否存在明显资源浪费:
- 【测试与回归检查】
  - 是否容易验证:
  - 是否需要补测试:
  - 是否存在明显回归风险:
- 【AI 生成风险检查】
  - 是否存在基于想象的接口/字段/工具:
  - 是否偷偷改需求:
  - 是否复制相似代码但上下文不适配:
  - 是否遗漏已有项目约束:
- 【问题列表】
  - 问题 1:
  - 严重性:
  - 证据:
  - 问题 2:
  - 严重性:
  - 证据:
- 【风险评估】
  - 残余风险 1:
  - 残余风险 2:

【约束】
- tester 必须覆盖主要风险和未决问题
- 不得把 review 通过当作功能正确的证明

【校验标准】
- 所有改动区域都已检查
- 发现项基于证据
- 即使无 bug 也写清残余风险

【禁止事项】
- 不静默改代码
- 不重写设计
- 不做空泛通过结论

【交接给下一个角色】
- 下一角色：测试员【负责验证正常路径、异常路径与边界情况】
- 可用输入：原始需求 + solution handoff + changed code + 本 handoff
- 非目标：直接改代码
- 完成条件：每个主要风险都有测试或阻塞说明
```
